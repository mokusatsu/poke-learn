# POKE-LEARN アプリケーション仕様書

本書は `poke-learn` プロジェクトの機能仕様、UI/UX の挙動、およびドメイン固有のロジック（クイズ出題アルゴリズム等）をまとめた仕様書です。

---

## 1. システム概要
本アプリケーションは、ポケモンの複雑なタイプ相性や実戦における選出（6vs6ドラフト）の思考プロセスを体系的に習得するための近未来的学習システムです。
React + Vite + Deno を基盤とし、状態管理はReactのローカルステート（`useState`等）を主体としたシンプルな構成を採用しています。

### メイン機能（タブ構成）
1. **🎯 タイプ相性 (TypeQuiz)**: 単一・複合タイプ間の基本的なダメージ倍率を答えるクイズ。
2. **⚔️ 6vs6 選出 (SelectionQuiz)**: 提示された相手パーティ（6匹）に対して、有利な自パーティ（3匹）を選出するクイズ。
3. **🔥 一貫技 (ConsistencyQuiz)**: 相手の複数匹のポケモン全員に対して、「半減以下にされない」攻撃タイプを見抜くクイズ。
4. **🌐 相性可視化 (TeamVisualizer)**: 任意のポケモン6匹で構成されたパーティの弱点・耐性をマトリクスで可視化するツール。
5. **📊 苦手分析＆特訓 (WeaknessAnalysis / MaxPowerQuiz)**: 過去の解答履歴から苦手なタイプ相性を抽出し、反復特訓を行うモード。シャドウボス戦闘画面におけるボスのタイプ表示は、他のゲームモードと同様に正式な `TypeBadge` コンポーネントを用いたバッジ形式で統一表示される。
6. **奥付 (Colophon)**: ヘッダーのリリース日バッジ（`2026-06-25`）をクリックすることで遷移。システムのコピーライト（`COPYRIGHT.md`）および開発履歴（`HISTORY.md`）をエレガントに表示する。

## 2. 各クイズ機能の詳細仕様

### 2.1 タイプ相性 (TypeQuiz)
* **目的**: 攻撃側と防御側（単一・複合）のタイプ相性（弱点、耐性、無効）を直感的に解答できるようにする。
* **学習の段階的アンロックと知識伝播（DP）モデル**:
  * 最初は基本的な3タイプ（ほのお、みず、くさ）のみが出題される。
  * ユーザーの進捗や設定に応じて、メジャー度順にタイププールが徐々に拡張され、最大18タイプまで段階的に学習できる。
  * **実効カバー率とアンロックの緩和（知識伝播）**:
    * 段階的学習を次に進める（アンロック）ための網羅率（カバー率）計算には、直接回答した履歴から「対称性」および「3すくみ循環（サイクル）」のルールに基づいて知識の伝播を推論（不動点計算）する。これによってすべてのペアを網羅しなくても次に進めるよう学習が緩和される。
    * **対称性伝播**: $A \to B$ が非等倍（特徴的）かつカバーされ、逆の $B \to A$ も非等倍で、かつ両者の相性が対称的（同じ倍率、または共に非等倍）な場合、逆もカバーとみなす。
    * **3すくみ循環伝播**: 3つのタイプ間で循環的な強弱（特徴的相性）関係があり、そのうち2つを直接カバーした場合、残りの1つのペアもカバーとみなす。
    * **非対称ペアの除外**: 行きと帰りで効果が対称になっていない非対称ペア（例：ノーマル $\to$ いわ（半減）/ いわ $\to$ ノーマル（等倍）など、片方のみが等倍のペア）は、推論によるカバーの対象外となり、等倍であっても必ず直接回答してカバーされなければならない。
    * **等倍自動カバー**: 特徴的相性のペアがすべてカバーされ、かつ未カバーの非対称ペアがプール内に存在しない状態に達した時点で、残りの対称な等倍ペア（例：ノーマル $\to$ ノーマルなど）もすべて自動的にカバーされたとみなし、網羅率100%となって次のレベルを解放する。
  * **非対称ペアの優先出題（逆ペナルティ）**:
    * 混同しやすい非対称ペアについて、まだカバーされていない場合はクイズ出題時の選出ウェイト（出現確率）が通常の **2倍** に引き上げられ、優先的に出題される。
  * **アンロック時の強制遷移防止**: カバー率が100%になり次のタイプが自動解放される際、回答結果の確認ができるよう、回答済み状態（`isAnswered === true`）のうちは自動での再出題を一時保留する。ユーザーが「次の問題へ」をクリックした時点で初めて、新しいタイプを含んだ問題が生成される。手動でレベルが変更された場合は即座に問題がリセットされる。
* **出題カテゴリ**:
  * シンプル相性（4択: 2倍, 等倍, 半減, 無効）：選択肢をクリックした瞬間に回答が確定するため、回答決定前の状態では画面下部の「回答を決定する」ボタンは非表示となる。
  * 単一/複合に対する攻撃（該当する弱点タイプをすべて選択）
  * 単一/複合に対する防御（該当する耐性タイプをすべて選択）
* **重複回避制約**:
  * 同じ問題が連続して出題されるのを防ぐため、直前の出題キー（カテゴリとターゲット/ペアの組み合わせ）を保持する。新しく抽選された問題が直前と同じ場合は、最大20回まで再抽選を行う。
* **苦手克服モード**: `localStorage` (`poke-learn-quiz-weights`) に記録された誤答履歴に基づき、間違えやすいタイプ・ペアの出現確率を動的に高める。

### 2.2 6vs6 選出 (SelectionQuiz)
* **目的**: 実戦のパーティ単位での選出思考（誰を通して誰で受けるか）を養う。
* **クイズカテゴリ**:
  1. **offense-4x (4倍弱点攻撃選出)**: 相手の4倍弱点を持つポケモンすべてに対し、4倍弱点(STAB)を突けるポケモンを3匹選ぶ。
  2. **defense-4x (4倍回避防御選出)**: 相手のどの攻撃タイプからも4倍弱点を受けない安全な3匹を選ぶ。
  3. **offense-all (一貫性攻撃選出)**: 選出した3匹のタイプ一致技で、相手6匹全員に対して「2倍または4倍」の弱点を突ける組み合わせを選ぶ。
  4. **defense-min-2x (鉄壁防御選出)**: 4倍弱点を受けず、かつ「2倍弱点を受ける味方の数」を最小に抑える組み合わせを選ぶ。
* **構成的ドラフトシステム**: ランダム生成時に「正解となる選出が存在しない」ケースを防ぐため、特定の弱点を持つ相手に対しては確定でカウンターとなる味方をプールに混ぜ込むロジックが組み込まれている。

### 2.3 一貫技 (ConsistencyQuiz)
* **目的**: 相手のパーティ（3匹または6匹）全員に対して、「等倍（1.0倍）以上で通る（＝半減・無効化されない）」攻撃タイプを見抜く能力を養う。
* **出題ロジック**:
  * 相手チームをランダム生成する際、全18タイプのうち「正解（一貫するタイプ）」が1〜9個になるまでリトライを行う（すべてが一貫するような極端なケースを除外）。
* **苦手克服連携**: `poke-learn-consistency-weights` の履歴から、もっとも苦手とするタイプを相手パーティの先頭に意図的に組み込む。

### 2.4 最大打点クイズ / 特訓モード (MaxPowerQuiz)
* **目的**: 特定のポケモンに対して最も大きなダメージ（最大倍率）を与えられるタイプを素早く選択する。
* **難易度仕様**:
  * **Normal**: 最も高い倍率（2.0倍 または 4.0倍など）のタイプを正解とする。
  * **Expert**: 直感に反する複雑な複合タイプ計算を要求する。
* **Expert モードの特別ルール (後半2問の強制出題)**:
  * 2倍や4倍の弱点が存在する場合でもそれらは正解の選択肢から**除外されない**。
  * ただし、クイズの後半（最終2問など）において、**「0.5倍 × 2.0倍 = 1.0倍（等倍）」が最大打点（正解）となるような、非常に難解な複合タイプパターン**を強制的に出題する。
  * *注: UIテキストで「0.5×2=1倍になるパターンが出ます」とは明示せず、実力で導き出させる仕様とする。*

### 2.5 相性の覚え方とイラスト画像表示仕様
ユーザーがクイズで間違えた際、または苦手分析マトリクスでホバーした際に、タイプ相性の論理的な「覚え方（考察）」および「視覚イラスト画像」を表示し、記憶の定着を支援する。
* **データ参照とフォールバック**:
  * 考察文および画像は `typeMatchupRationales` データを参照して引き当てる。
  * 画像は `public/images/matchup-rationales/<attacker>-<defender>.png` から動的に読み込む。
  * **画像存在チェック (onError)**: まだ画像が生成されていない相性ペアについて表示が崩れたり壊れた画像アイコンが表示されたりするのを防ぐため、`<img>` タグの `onError` イベントを検知し、画像が無い場合は画像表示エリア自体を自動で非表示とし、考察テキストのみを綺麗にレイアウトして表示する。
  * **画像と解説テキストのレイアウト最適化**: アコーディオン展開時の縦方向の占有スペースを最小限に抑えるため、PC/タブレット表示時は左側に「タイプ相性の矢印・倍率表示と考察テキスト」を縦方向にまとめ、右側に「イラスト画像」を横並びで配置する左右入れ替えレイアウトとした。また、スマホ等の狭い画面（767px以下）ではレスポンシブデザインにより自動で縦積み（上にテキスト、下に画像）へ切り替わるようにし、画像サイズも横幅100%（最大240px）に調整してアスペクト比を保ちつつ、縦並び時に画像の高さを自動調整（高さ制限のインライン flex-basis の排除）することで画像の一部が切り取られたりはみ出したりする表示崩れを防いでいる。
* **各クイズ画面での表示挙動**:
  * **タイプ相性 (TypeQuiz)**:
    * **次の問題へボタンの配置最適化**: 回答後に解説（相性の覚え方）や画像が展開された際、「次の問題へ」ボタンが解説の下に押し下げられてしまい操作のリズムが崩れるのを防ぐため、正誤結果を表示するフィードバックパネルの右側（インライン）にボタンを配置。解説の長さに関わらず常に同じ位置で快適に「次の問題へ」を操作できるレイアウトとした。
    * **モード切り替え時の状態リセット保証**: クイズカテゴリや苦手克服モードを切り替えた際、以前の正答・誤答表示や選択状態が画面に一瞬残るフラッシュバグを防ぐため、切り替え操作のイベントハンドラー内で回答状況・選択状態・正誤判定（`isCorrect`含む）を一括バッチ処理でリセットし、新しい問題へのリセットが単一の描画サイクルで同時に行われるようにした。
    * **シンプル相性モード**: 不正解時、結果表示エリアの下に自動で「💡 相性の覚え方」セクションが展開され、考察テキストと画像（あれば）を表示する。
    * **弱点クイズ (単一/複合) モード**: 不正解時、解答タイプバッジ（正解・誤答バッジ）のうち、**解説データ（考察）が存在するタイプにのみ「💡」アイコンが付与される**。バッジをクリックすることで、該当するタイプと出題タイプ間の相性考察および画像を表示するアコーディオンが開閉式で展開される。複合タイプの場合は、攻撃タイプから防御側のそれぞれのタイプ（2ペア分）の考察を並べて表示する。また、**回答結果フィードバックに列挙される正解のタイプも, テキストではなく視覚的なタイプカード（TypeBadge）で表示される。これらのカードもクリック可能であり、クリックすると同様にアコーディオンで相性考察が展開される（正解のタイプカードには「💡」マークは付与されない）**。
  * **6vs6 選出 (SelectionQuiz)**:
    * **次の対戦カードへボタンの配置最適化**: 回答後に表示される「選出解説」のテキスト量が多くなった際、「次の対戦カードへ」ボタンが押し下げられて操作テンポが崩れるのを防ぐため、結果・解説が表示されるフィードバックパネルの右側（インライン）にボタンを配置。解説の長さに依存せず一定の位置でスムーズに進められるレイアウトとした。
  * **最大打点クイズ (MaxPowerQuiz)**:
    * 不正解時、自己修正用に乗算計算プロセスが展開されるリスト内のタイプ名をクリックすることで、同様の「相性の覚え方」アコーディオンをそのすぐ下に展開表示する。
  * **一貫技クイズ (ConsistencyQuiz)**:
    * 不正解時、解説欄に表示される「一貫を防いだ要因（半減以下にした相手ポケモン）」の分析リストの行、または相手のポケモンカードをクリックすることで、そのポケモンと防がれた攻撃タイプの相性考察アコーディオンを動的に展開表示する。
  * **弱点分析 (WeaknessAnalysis)**:
    * 弱点・耐性マトリクスの各セル（攻撃側 ➡ 防御側）にマウスホバーした際、ブラウザ標準のツールチップテキスト（title属性）の改行下に、追加で `【考察】: 考察文章` をテキストのみで表示する（画像は表示しない）。

## 3. 共通・ドメインロジックの仕様

### 3.1 タイプ相性倍率 (utils/typeMatrix.ts)
* **倍率の定義**: 4.0倍, 2.0倍, 1.0倍, 0.5倍, 0.25倍, 0.0倍（無効）の6段階。
* **フレーバーテキスト表記統一**:
  表記の揺れ（「抜群」「バツグン」等）を防ぐため、ドキュメントおよびUI上では以下の平仮名/カタカナ表記に厳格に統一する。
  * 2.0倍 / 4.0倍: 「ばつぐん」
  * 0.5倍 / 0.25倍: 「いまひとつ」
  * 0.0倍: 「こうかがない」
* **由来と考察のドキュメントおよびデータ同期**:
  各タイプ相性の論理的・現実世界的な理由や考察については、[`docs/type-matchup-rationale.md`](./docs/type-matchup-rationale.md) に全18タイプ分の解説が網羅されている。このドキュメントの記述は、プログラムから動的に取り扱うため、[typeMatchupRationales.ts](file:///C:/Users/lain/.gemini/antigravity/worktrees/poke-learn/dynamic-prog-learning-logic/src/data/typeMatchupRationales.ts) に抽出されたレコードデータと常に完全に同期対象となっている。

### 3.2 PokeAPIとキャッシュ戦略
* ポケモンの基本データ（ID、名前、タイプ）は PokeAPI (`pokeapi.co`) から取得する。
* 取得したデータは `localStorage` に `poke-learn-cache-[id]` の形式で永続化される。
* オフライン時でもキャッシュを利用してクイズが機能するよう、`App.tsx` にて `navigator.onLine` を監視し、ヘッダーにオンラインステータスとキャッシュ件数を表示する。

### 3.3 UIパフォーマンスのガイドライン
* ホバー時の光彩（Glow）エフェクトや微小なアニメーションなど、高頻度なUI更新に対しては React の `useState` を用いず、Vanilla CSSの擬似クラス（`:hover`, `:active`）および `transform`, `opacity`, `will-change` を活用して GPU アクセラレーションを効かせた実装を行う。

### 3.4 モバイル表示（スマートフォン縦画面）のUI最適化
スマートフォン縦画面（`isMobile`判定）では、クイズ・コンテンツエリアの表示領域を確保するため、上部ナビゲーションの高さを極力削減するレスポンシブデザインを適用する。
* **メインタブの1行スクロール化**: メインタブを折り返しなし（`flexWrap: "nowrap"`）で横スクロール可能（`overflowX: "auto"`）な1行に固定し、ラベルテキストを短縮形（例：「🎯 タイプ相性」→「🎯 相性」、「📊 苦手分析＆特訓」→「📊 分析・特訓」）に切り替える。
* **サブカテゴリのセレクトボックス（ドロップダウン）化**: `TypeQuiz` や `SelectionQuiz` の出題形式/カテゴリ選択ボタン群は、スマホ表示時に絵文字付きの1つの `<select>` 要素に自動で置き換えて高さの占有を1行分に抑える。
* **段階的学習コントロールパネルのポップオーバー化**: `TypeQuiz` の段階的学習設定エリアは、常時表示を「レベルと網羅率」を示す1枚のバッジボタンに集約する。設定ボタンがタップされた際のみ、進捗の操作やリセットが可能なインラインポップオーバー（設定パネル）を展開する。
* **リザルト画面解答履歴の2行化**: `MaxPowerQuiz` の解答結果画面（10問履歴リスト）において、スマホ表示時は1行あたりの情報詰め込みによる文字の縦折れを防ぐため、フレックス方向を縦並び（`flexDirection: "column"`）とし、問題番号・ポケモン名と、詳細データ（手札、選択、時間、正誤）を2行に分けて表示する。
* **最大打点回答画面の重なり防止**: `MaxPowerQuiz` のクイズ回答画面において、スマホ表示時に接続先タイプバッジとポケモンカードが重なるのを防ぐため、スマホ表示用の仮想ポケモンカード幅（`POKEMON_CARD_WIDTH`）を `96px` から `112px` に拡張し、配置およびSVG結線の終点座標（`x2`）を左側に寄せることで適切な隙間を確保する。
* **選出画面カードハイライト時の見切れ防止**: `SelectionQuiz` の敵・味方チームの横スクロールコンテナにおいて、スマホ表示時に選択カードが拡大（`scale`）および発光（`box-shadow`）した際に親コンテナの境界で切り取られるのを防ぐため、スマホ用コンテナに内側余白（`padding: 10px 6px`）を十分に確保する。
* **最大打点「次へ」ボタンの配置最適化**: `MaxPowerQuiz` のクイズ回答画面において、回答後に解説が表示された際、スマホ表示時に「次の問題へ」ボタンが画面外（最下部）に隠れてスクロールが必要になるのを防ぐため、ボタンを右側の相手ポケモンカードの直下に「次へ ➡」としてコンパクトに再配置する。同時に画面最下部の重複ボタンを削除する.
* **タイプ相性「次の問題へ」ボタンのインライン化と折り返し対応**: `TypeQuiz` の正誤フィードバックパネル内に「次の問題へ」ボタンをインライン配置。スマホなどの極端な狭画面時には自動で折り返して下部に落ちるレスポンシブな `flex-wrap: "wrap"` レイアウトを適用することで、表示崩れやクリッピングを防ぎつつ、快適な操作性を確保した。
* **相性可視化の縦方向レイアウト化とタイプ表示**: `TeamVisualizer` において、スマホ表示時は左右2カラム構成から、上部：味方（横並び）、中央：接続線（上から下への流線）、下部：相手（横並び）の縦方向フローへとレイアウトを自動切り替えする。また、狭い横幅でも各ポケモンのタイプを識別できるよう、画像の直下に極小の日本語1文字ドット（例：「火」「水」「草」等）を表示する。
* **スマホでのタップ選択トグル化**: マウスホバーが利用できないスマホ端末に対応するため、ポケモンカードをタップ（クリック）することで選択状態（ハイライトおよび関係線の表示）をトグル（ON/OFF）するイベントを実装。別のカードをタップすると切り替わり、カード外の余白部分をタップした際にはハイライト選択を全面的にクリアする仕様とした。

## 4. アカウント・同期機能の仕様

### 4.1 アカウントの自動作成と管理
* **初回アクセス時の自動作成**: アプリケーションを初めて開いた際、サーバーAPI (`/api/account/create`) を介してランダムなユーザー名（例：`Trainer-XXXX`）と、引き継ぎ・復元用の暗号論的に安全なUUID形式の「引き継ぎハッシュ」が自動で生成され、ローカルストレージに保存される。
* **ヘッダーへの表示**: ヘッダー右上に 👤 アイコンとユーザー名を表示。クリックすることで隠しタブである「👤 アカウント設定」へ遷移する（スマホ表示時はユーザー名は非表示で 👤 アイコンのみ表示）。
* **ユーザー名の変更**: アカウント設定画面にてユーザー名を変更（1〜20文字）可能。変更した内容は即座にサーバーに反映される。

### 4.2 学習進捗データの自動同期
* **同期対象データ**:
  * ポケモンの基本データキャッシュ（PokeAPI キャッシュ）以外の、すべての学習進捗・回答履歴データ（誤答重みウェイト、カバー済みペア、解放レベル、勲章メダル一覧等）。
* **透過的な同期トリガー（モンキーパッチ）**:
  * `localStorage.setItem` および `localStorage.removeItem` の実行をグローバルにパッチ（インターセプト）し、同期対象のキーが更新された場合に自動で同期を検知。
  * 各クイズ画面の React コードを変更することなく、透過的な自動同期を実現している。
* **デバウンス処理**:
  * 頻繁な回答による過剰なAPI通信を防ぐため、変更検知後 **3秒** のデバウンス（遅延実行）を挟んでから、まとめて `/api/account/sync` へ同期リクエストを送信する。

### 4.3 アカウントの復元と引き継ぎ
* **引き継ぎ用ハッシュ**:
  * アカウント設定画面で引き継ぎ用ハッシュを表示（デフォルトではセキュリティのためマスク表示、目のアイコンクリックで可視化、コピーボタンでクリップボードへコピー可能）。
* **復元の実行とリセット**:
  * 別マシンのハッシュを入力して「アカウントデータを復元する」ボタンを押すと、確認の警告ポップアップ（データが上書きされる警告）が表示され、同意後に復元が実行される。
  * 復元成功時は、ローカルの `localStorage` が完全に上書きされた後、安全に進捗状態を反映させるため、ブラウザが自動的にリロード（再読み込み）される。

### 4.4 Deno KV バックエンドと期限管理 (TTL)
* **データの有効期限 (1年)**:
  * すべてのユーザーアカウントおよび進捗データは、Deno KV 内に **1年間** の有効期限 (`expireIn: 31,536,000,000` ms) を設定して保存される。
  * データの新規作成、取得、同期、ユーザー名変更などのアクセスが発生するたびに、有効期限は1年後に自動で再延長される。最後のアクセスから1年間全くアクセスのなかったアカウントデータは、自動的に完全に削除される。
* **ローカルテスト時のデータベースバインド**:
  * Deno サーバーがローカルテスト環境で動いている場合は、デフォルトで Database ID を `6721ce--local` に振り向けて接続する。
  * 環境変数や起動オプション等の明示的なブランチ指定がある場合は、Git の現在ブランチ名などを自動取得して `6721ce--<branch_name>` に動的に接続する柔軟性を備える。
  * 本番環境 (Deno Deploy) 上では、引数なしの `Deno.openKv()` によりプラットフォーム上のデータベースに自動バインド接続される。


### 4.5 デプロイ構成と静的ファイルホスティングの仕様
* **同一ドメインホスティング (Deno Deploy)**:
  * `server.ts` が Web サーバーのエントリーポイントとして機能し、APIエンドポイント以外のすべての静的ファイルリクエスト（HTML, CSS, JS等）を `dist` ディレクトリから配信する機能を内包する。
  * ファイルが見つからない（404）場合は、SPA (Single Page Application) のルーティングを維持するため自動的に `dist/index.html` にフォールバックして配信する。これにより、Deno Deploy 単体でフロントエンドとバックエンドを同一ドメインで同居ホストできる。
* **別ドメインホスティング（マルチホスト構成）**:
  * フロントエンド（静的ホスティング）とバックエンド（Deno Deploy）が別々のホストに分離されている構成にも対応するため、環境変数 `VITE_API_BASE_URL` をサポート。
  * クライアントは `VITE_API_BASE_URL` で設定されたベースURL（例: `https://poke-learn-api.deno.dev`）に対して API リクエスト（CORSプリフライト対応）を送信する。未指定時は相対パス `/api/...` が使用される。
