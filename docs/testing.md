# テスト戦略 (Testing Strategy)

本プロジェクトでは、アプリケーションの信頼性を確保するため、主に **Playwright** を利用した E2E (End-to-End) テストを実施しています。

## 1. テストフレームワーク

- **Playwright**: `@playwright/test` を使用し、実際のブラウザ環境でレンダリングやレイアウトが崩れていないかを検証します。

## 2. テストの実行方法

Deno 環境下で npm のパッケージを実行するため、以下のコマンドを使用します（`npm run` ではありません）。

```powershell
# E2Eテストの実行
deno run -A npm:@playwright/test test
```

> **注意**: 開発環境が Windows であるため、上記コマンドは PowerShell またはコマンドプロンプト上で実行されます。

## 3. テストファイルの配置と命名規則

- テストファイルは `tests/` ディレクトリ直下に配置します。
- ファイル名は `*.spec.ts` という形式にします。
  - 例：`tests/viewport-layout.spec.ts`

## 4. 主なテスト対象と方針

本プロジェクトの特性上、UIのレイアウトやレスポンシブデザインが重要な要素となっています。したがって、以下の点に重点を置いてテストを記述してください。

- **ビューポートごとのレイアウトテスト**:
  - `WeaknessAnalysis` (相性マトリクス) のような複雑な UI コンポーネントが、モバイルからデスクトップまで異なる画面幅（Viewport）で正しく表示されること。
  - CSSによるレスポンシブな折り返しや、はみ出し（オーバーフロー）が発生していないかの検証。
- **機能テスト**:
  - 各クイズコンポーネントにおける正誤判定ロジックの動作検証。
  - PokeAPI やローカルデータが正しく読み込まれ、UI にバインディングされていることの検証。

## 5. エージェント向け：テスト追加時のガイドライン

エージェントが新機能や大きな UI 変更を行った場合は、必ず `tests/` 配下の既存のテストを実行し、デグレ（既存機能の破損）が起きていないか確認してください。必要であれば新しい `*.spec.ts` を追加し、テストカバレッジを維持・向上させてください。
